## 一、合同规则

### 1.1 合同类型

FinanceDesk 仅涉及两种合同类型：

| 类型 | 定义 | 特点 |
|------|------|------|
| **单项合同** | 一个合同对应一个项目，一一绑定 | 合同金额 = 项目预算，无框架概念 |
| **框架合同** | 一个合同框架下可挂载多个订单 | 先签订框架协议，后按具体项目下达订单 |

### 1.2 生命周期

```
框架合同签订 → 项目创建 → 订单下达 → 订单执行 → 订单完成
                                                   ↓
                             收入流水 → 回款 → 成本流水 → 支付
```

- 框架合同是项目生命周期顶层文档，不直接参与财务核算
- 项目的 `framework_name` 字段记录所属框架合同名称
- 供应商框架合同 (`SupplierContract`) 独立管理，不与项目直接绑定

### 1.3 与订单的关系

| 合同类型 | 订单绑定方式 |
|---------|-------------|
| 单项合同 | 订单直接绑定项目，一个项目可多个订单 |
| 框架合同 | 订单先绑定项目，项目再关联框架合同 |

### 1.4 合同数据来源

```
合同名称        → 人工录入（项目创建时填写 framework_name）
供应商框架合同  → Excel 导入（合同导入模板.xlsx）
框架编号        → 人工录入或 Excel 导入
合同金额        → 人工录入或 Excel 导入
```

## 二、订单规则

### 2.1 唯一经营结算单元

> **订单是整个系统的唯一经营结算单元。所有经营数据最终必须归属于订单。**

这条规则是整个 FinanceDesk 业务模型的基石：

```
项目（经营容器）
  └── 订单（结算单元）
        ├── 收入流水（开票） → 回款（收款）
        └── 成本流水（支出） → 支付（付款）
```

### 2.2 数据归属规则

| 数据类别 | 归属层级 | 说明 |
|---------|---------|------|
| 收入 | 订单级 | IncomeFlow.order_id FK → Order |
| 成本 | 订单级 | CostFlow.order_id FK → Order |
| 开票金额 | 订单级 | 来自 IncomeFlow，按 order_id 聚合 |
| 回款 | 订单级 | 来自 Collection，通过 IncomeFlow → Order 追溯 |
| 支付 | 订单级 | 来自 Payment，通过 CostFlow → Order 追溯 |
| 订单利润 | 订单级 | 系统自动计算（收入合计 - 成本合计） |

### 2.3 禁止出现无法归属订单的数据

- 所有收入流水（IncomeFlow）必须有 `order_id` 外键约束（物理 FK）
- 所有成本流水（CostFlow）必须有 `order_id` 外键约束（物理 FK）
- 回款（Collection）通过 IncomeFlow → Order 溯源
- 支付（Payment）通过 CostFlow → Order 溯源
- **不得存在游离于订单之外的收支数据**

### 2.4 订单字段规范

| 字段 | 规则 | 说明 |
|------|------|------|
| `order_no` | 全局唯一 | 订单编号，UNIQUE 约束 |
| `project_id` | 必填，物理 FK | 订单必须归属项目 |
| `supplier_id` | 可选，物理 FK | 订单必须关联供应商 |
| `amount` | 必填，默认 0 | 含税金额 |
| `non_tax_amount` | 可选 | 不含税金额 |
| `status` | 字典化 | 待执行 / 执行中 / 已完成 / 已作废 |

### 2.5 订单数据来源

```
订单编号        → 人工录入 或 ERP 导入
订单金额        → 人工录入 或 ERP 导入
订单状态        → 人工维护 或 系统自动计算
```

### 2.6 物理外键约束（实际代码验证）

经代码审查确认，Order 模型使用物理外键（非逻辑关联）：
- `project_id` → `ForeignKey("project.id", ondelete="RESTRICT")`
- `supplier_id` → `ForeignKey("supplier.id", ondelete="RESTRICT")`

## 三、数据来源规则

### 3.1 来源三分类

系统中所有字段的数据来源必须归属以下三类之一：

| 来源 | 标识 | 说明 |
|------|------|------|
| **人工维护** | `manual` | 通过前端 CRUD 或 Excel 导入人工录入 |
| **ERP 导入** | `erp` | 来自智慧工程平台的自动同步数据 |
| **系统自动计算** | `system` | 系统根据已有数据自动计算的结果 |

### 3.2 来源标注规则

- **每个业务对象**的所有字段必须可追溯其来源
- **不在三分类中的字段来源不得存在**
- 来源标注方式：
  - 后端模型字段：通过 `source_type` 字段标注（如 BudgetAdjustment）
  - 前端显示：在台账/详情中显示数据来源标记
  - 导入数据：Excel 导入标记为 `manual`，ETL 导入标记为 `erp`

### 3.3 现有模型来源追溯

| 模型 | source 字段 | 说明 |
|------|-----------|------|
| Project | 无独立 source | 全字段人工维护，来源明确 |
| Supplier | 无独立 source | 全字段人工维护或 Excel 导入 |
| Order | `erp_no` 隐式标注 | 有 erp_no 的记录标记为来自 ERP |
| IncomeFlow | 无独立 source | 人工录入或 Excel 导入 |
| CostFlow | 无独立 source | 人工录入或 Excel 导入 |
| Collection | 无独立 source | 人工录入，来源明确 |
| Payment | 无独立 source | 人工录入，来源明确 |
| BudgetAdjustment | `source_type` ✅ | 已有来源追溯字段（order/income_flow/cost_flow/manual） |

### 3.4 未来要求

> **任何新增字段必须同时注明数据来源。**

## 四、ERP 规则

### 4.1 ERP 是事实来源

- **智慧工程平台**是 FinanceDesk 所有业务数据的权威来源
- ERP 中已有的数据，FinanceDesk 不重复录入
- ERP 数据不可修改（只读）

### 4.2 双轨同步架构

```
ERP 系统 ──→ 暂存表（Staging）──→ ETL 清洗 ──→ 业务表
                    ↑
            人工录入 ──────────────→ 业务表（不受影响）
```

- **不修改现有业务表结构**
- **新数据走暂存表隔离**
- **业务表始终保持手工可写**
- 同步过程可重放、可追溯

### 4.3 FinanceDesk 不重复记账

| 场景 | ERP 负责 | FinanceDesk 负责 |
|------|---------|-----------------|
| 项目立项 | 智慧工程平台 | 同步项目信息 |
| 订单下达 | 智慧工程平台 | 同步订单信息 |
| 开票记录 | 财务系统 | 同步开票数据 |
| 回款确认 | 银行系统 | 同步回款数据 |
| 人工台账 | 不负责 | Excel 导入补充 |
| 经营分析 | 不负责 | Dashboard 汇总 |

### 4.4 数据不可修改规则

- ERP 同步到 FinanceDesk 的数据标记 `source=erp`
- ERP 来源的字段在前端显示为只读
- 如需修正 ERP 数据，必须在 ERP 侧修改后再同步
- FinanceDesk 不提供 ERP 来源字段的手工修改入口

### 4.5 当前 ERP 集成状态

| 模块 | 集成状态 | 匹配键 | 同步方式 |
|------|---------|--------|---------|
| 项目 | ✅ ETL 已完成 | `erp_no` | 增量 |
| 订单 | ✅ ETL 已完成 | `order_no` | 增量 |
| 收入流水 | ✅ ETL 已完成 | `invoice_no` | 增量 |
| 成本流水 | ✅ ETL 已完成 | 模糊匹配 | 增量 |
| 生产部署 | ⏳ 待定 | — | — |

## 五、Dashboard 规则

### 5.1 只读规则

> **Dashboard 不允许录入任何数据。Dashboard 只能读取和展示。**

### 5.2 允许读取的数据源

| 数据来源 | 说明 |
|---------|------|
| 合同数据 | 项目信息、供应商框架合同 |
| 订单数据 | 订单列表及明细 |
| 流水数据 | 收入流水、成本流水汇总 |
| 回款/支付数据 | 回款汇总、支付汇总 |
| ERP 同步数据 | 来自智慧工程平台的同步数据 |
| 系统计算结果 | 利润分析、应收账龄、缺口计算等 |

### 5.3 允许的展示范围

| Dashboard 端点 | 展示内容 | 数据来源 |
|---------------|---------|---------|
| `/project-summary` | 项目维度的订单数、收入、回款、应收余额 | 系统自动聚合计算 |
| `/project-profit` | 项目利润分析 | 系统自动计算（收入-成本） |
| `/order-detail/{id}` | 订单维度的开票、回款、成本、支付 | 系统自动聚合 |
| `/ar-aging` | 应收账龄分析 | 系统自动计算 |

### 5.4 禁止行为

- ❌ Dashboard 页面禁止出现任何表单输入、数据提交
- ❌ Dashboard 禁止修改数据库中的任何记录
- ❌ Dashboard 禁止触发任何数据录入流程
- ❌ Dashboard 禁止维护状态（不是状态机）

### 5.5 Gap 计算规则

- Gap（差额）永远由系统自动计算，不允许手工修改
- Gap 计算方式：`应收余额 = 开票金额合计 - 回款金额合计`
- Gap 计算方式：`应付余额 = 成本金额合计 - 支付金额合计`

## 六、供应商规则

### 6.1 重新定义：年度成本供应商合同库

> 供应商模块不是客户关系管理（CRM），而是**年度成本供应商合同库**。

### 6.2 一份合同 = 一条记录

| 规则 | 说明 |
|------|------|
| 一条记录 = 一个供应商的一个框架合同 | 不允许"一个供应商维护多个年度" |
| 年度变化 → 新合同记录 | 每年有新的框架合同需要重新创建记录 |
| 供应商名称 | 唯一标识，同一法律主体不可重复 |
| 框架编号 (`framework_no`) | 必填，唯一标识合同 |

### 6.3 供应商模型（实际代码中的表结构）

| 表名 | 说明 | 与 Supplier 关系 |
|------|------|-----------------|
| `supplier` | **主表** — 供应商 + 框架合同合并 | — |
| `supplier_contract` | 框架合同补充信息 | FK → supplier.id |
| `supplier_price` | 旧单价表（兼容已有数据） | FK → supplier.id |
| `supplier_year_price` | 年度综合单价（UQ: supplier_id+year） | FK → supplier.id |
| `supplier_unit_price` | 按年度版本的新单价 | FK → supplier.id |

### 6.4 供应商数据来源

```
供应商名称        → 人工录入 或 Excel 导入（供应商导入模板.xlsx）
框架编号          → 人工录入 或 Excel 导入
合同信息          → Excel 导入（合同导入模板.xlsx）
价格信息          → Excel 导入（单价导入模板.xlsx）
年度单价          → Excel 导入或人工维护
```

## 七、Excel 规则

### 7.1 必须支持导入导出

> **所有基础数据模块必须支持：统一 Excel 导入 + 统一 Excel 导出。**

### 7.2 Excel 导入要求

| 项 | 必须 | 说明 |
|----|------|------|
| 模板统一 | ✅ | 每个模块一套标准模板，模板列头用中文 |
| 批量导入 | ✅ | 一次导入最少支持 1000 行 |
| 去重机制 | ✅ | ON CONFLICT DO NOTHING |
| 错误反馈 | ✅ | 导入后返回成功/失败行数及错误详情 |
| 替换模式 | ✅ | 可选覆盖已有数据（upsert） |

### 7.3 Excel 导出要求

| 项 | 必须 | 说明 |
|----|------|------|
| 完整导出 | ✅ | 支持全部数据行导出 |
| 筛选导出 | ✅ | 支持按条件筛选后导出 |
| 模板一致 | ✅ | 导出的列头与导入模板保持一致 |

### 7.4 当前支持的导入模板（9 种）

| # | 模板 | 目标表 | 实现状态 |
|---|------|--------|---------|
| 1 | 项目导入模板.xlsx | project | ✅ |
| 2 | 供应商导入模板.xlsx | supplier | ✅ |
| 3 | 订单导入模板.xlsx | order | ✅ |
| 4 | 收入流水导入模板.xlsx | income_flow | ✅ |
| 5 | 成本流水导入模板.xlsx | cost_flow | ✅ |
| 6 | 回款导入模板.xlsx | collection | ✅ |
| 7 | 付款导入模板.xlsx | payment | ✅ |
| 8 | 合同导入模板.xlsx | supplier_contract | ✅ |
| 9 | 单价导入模板.xlsx | supplier_unit_price | ✅ |

### 7.5 导入技术规范

| 规范 | 说明 |
|------|------|
| 引擎 | pandas + openpyxl |
| 列映射 | 中文列头 → 英文字段（`translate_headers()`） |
| 日期处理 | `str → date.fromisoformat()` |
| 金额分流 | 贷方→收入，借方→成本；单列符号正数→收入，负数→成本 |
| 性能 | 矢量化切片，禁止 `iterrows()` |
| 架构 | 不修改现有业务表，新数据走暂存表 |

## 附录 B：术语表

|------|------|
| 项目 (Project) | 合同级业务容器，对应一个框架合同或单项合同 |
