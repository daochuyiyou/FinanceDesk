## BADR-001：FinanceDesk 产品定位

### 背景

FinanceDesk 已完成 M0-M3 工程治理、RC 验证和 Beta 启动。需要明确系统的最终业务定位，确定系统边界，避免功能蔓延。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 替代型 ERP | 完全替代智慧工程平台 ❌ 超出能力范围 |
| B | OA + 审批流 | 加入流程审批功能 ❌ 与财务分析目标无关 |
| C | 项目管理系统 | 以项目为中心的全生命周期管理 ❌ 与经营分析目标偏移 |
| D | **经营分析与结算管理平台** | 聚焦财务数据整合、分析和结算跟踪 ✅ |

### 最终决策

> **FinanceDesk 是施工企业经营分析与结算管理平台。**

### 决策原因

1. 核心用户需求是**替代手工台账**，不是替代集团级 ERP
2. 核心价值在于**经营分析**（Dashboard、利润分析、应收账龄）
3. 系统已有清晰的财务数据链路：项目→订单→流水→回款/支付
4. 明确边界有助于阻止无休止的功能新增

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无（不新增表） |
| API | 无 |
| 页面 | 无 |
| ERP | 确认智慧工程平台为事实来源，FinanceDesk 不替代 |
| Dashboard | 定位于分析层，不是录入层 |

### 后续不得违反事项

1. 不得添加 OA / 流程审批 / 人事 / 资产管理等功能
2. 不得尝试替代智慧工程平台
3. 所有新增功能必须评估是否符合"经营分析与结算管理"定位
4. 功能提议不符合定位的，直接拒绝

## BADR-002：合同模型

### 背景

系统中存在项目（Project）和供应商合同（SupplierContract）两类合同概念，但界限不清晰。需统一合同模型，明确仅允许两种合同类型。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 仅单项合同 | 过于简单，无法覆盖框架合同场景 ❌ |
| B | 仅框架合同 | 无法处理独立项目场景 ❌ |
| C | **单项合同 + 框架合同** | 覆盖全部实际业务场景 ✅ |
| D | 支持任意自定义合同类型 | 过于灵活，缺乏约束 ❌ |

### 最终决策

> **合同仅允许两种类型：单项合同、框架合同。禁止新增第三种合同类型。**

- **单项合同**：一个合同对应一个项目，一一绑定，合同金额 = 项目预算
- **框架合同**：一个框架协议下可挂载多个项目/订单

### 决策原因

1. 中移建设崇左分公司的实际业务仅涉及这两种合同场景
2. 两种类型的生命周期、订单关系、财务核算方式均不同，需分别处理
3. 增加第三种合同类型将引入不必要的复杂度，且无真实业务支撑

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无（现有 Project + SupplierContract 模型已覆盖） |
| API | 无 |
| 页面 | 无 |
| ERP | 无 |
| Dashboard | 无 |

### 后续不得违反事项

1. 不得新增第三种合同类型
2. Project.framework_name 字段仅用作记录合同名称，不承担业务逻辑
3. SupplierContract 的 contract_no 必须唯一

- [04_Data_Model.md](./04_Data_Model.md) 第 2.1 节（项目模块）

## BADR-003：订单模型

### 背景

订单（Order）是系统中数据关联最密集的实体。需要确认订单作为唯一经营结算单元的地位，确保所有经营数据最终归属订单。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 项目作为结算单元 | 项目粒度过粗，一个项目下多个订单的利润无法区分 ❌ |
| B | **订单作为结算单元** | Order 是所有收支数据的归属层级 ✅ |
| C | 流水作为结算单元 | 流水粒度过细，聚合分析困难 ❌ |

### 最终决策

> **订单是唯一经营结算单元。所有经营数据最终必须归属订单。**

归属规则：

| 数据类别 | 归属路径 |
|---------|---------|
| 收入 | IncomeFlow.order_id FK → Order.id |
| 成本 | CostFlow.order_id FK → Order.id |
| 回款 | Collection → IncomeFlow → Order |
| 支付 | Payment → CostFlow → Order |
| ERP 对账 | erp_no 字段 → Order |
| Dashboard 指标 | 系统自动按订单聚合 |

### 决策原因

1. 订单是财务核算的最小业务单元，收入和成本均在订单层级记录
2. 订单上有项目的 project_id 和供应商的 supplier_id，承担了链接项目和供应商的桥梁作用
3. 所有 Dashboard 聚合查询均基于订单聚合（project-summary/order-detail/project-profit）
4. 物理外键（`IncomeFlow.order_id → Order.id` 等）已在代码层确保归属

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无（现有 FK 约束已支持） |
| API | 无 |
| 页面 | 无 |
| ERP | ERP 同步必须以订单为单位 |
| Dashboard | 所有聚合以订单颗粒度为基础 |

### 后续不得违反事项

1. 不得创建不关联订单的收支数据
2. 不得允许游离于订单之外的经营数据存在
3. 新增任何数据表时，如需与财务核算关联，必须通过订单
4. 删除订单前必须检查是否存在关联的流水数据

- [Business_Rules.md](./Business_Rules.md) 第 2 节（订单规则）
- [04_Data_Model.md](./04_Data_Model.md) 第 2.3 节（订单模块）

## BADR-004：ERP 定位

### 背景

智慧工程平台是中移建设集团级 ERP 系统，FinanceDesk 与之存在数据重叠。需明确两套系统的边界和定位。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | ERP 替代型 | FinanceDesk 完全复制 ERP 功能 ❌ 重复投资 |
| B | **ERP 为事实来源** | FinanceDesk 同步 ERP 数据，不重复记账 ✅ |
| C | 双向同步 | FinanceDesk 可以修改并写回 ERP ❌ 违反集团管控 |

### 最终决策

> **ERP 是事实来源（Source of Truth）。FinanceDesk 不重复记账。ERP 数据只能导入，不允许修改。**

### 决策原因

1. 智慧工程平台是集团级系统，数据权威性和一致性由集团保障
2. FinanceDesk 是部门级经营分析工具，定位是补充而非替代
3. ERP 数据修改应由 ERP 侧流程控制，FinanceDesk 不应绕过
4. 双轨架构（暂存表隔离）已实现：ERP 数据→暂存表→清洗→业务表

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无（暂存表模式已实现） |
| API | 无 |
| 页面 | 无 |
| ERP | ERP 来源字段在前端显示为只读 |
| Dashboard | ERP 数据参与计算，但不可修改 |

### 后续不得违反事项

1. 不得在 FinanceDesk 中提供 ERP 数据的修改入口
2. 不得在 FinanceDesk 中重复录入 ERP 已存在的数据
3. ERP 数据更新必须走重新导入流程，不可手工修正
4. 导入的 ERP 数据必须标记 `source=erp`，与手工数据可区分

## BADR-005：数据来源

### 背景

系统中有多个数据入口（手工、Excel、ERP），数据可信度和维护责任不同。需要建立统一的数据来源分类规则，消除"来源不明"的数据。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 二分法：手工/自动 | 过于简单，无法区分 ERP 和系统计算 ❌ |
| B | **三分法：人工/ERP/系统** | 覆盖全部实际场景，分类清晰 ✅ |
| C | 多维分类 | 过于复杂，操作性差 ❌ |

### 最终决策

> **系统所有字段必须属于且仅属于以下三类：人工维护、ERP 导入、系统自动计算。**

| 来源类型 | 标识 | 说明 | 示例 |
|---------|------|------|------|
| 人工维护 | `manual` | 通过前端 CRUD 或 Excel 导入手工录入 | 项目名称、订单备注 |
| ERP 导入 | `erp` | 来自智慧工程平台的自动同步 | erp_no、ERP 订单金额 |
| 系统自动计算 | `system` | 系统根据已有数据自动计算 | Gap、利润、回款率 |

### 决策原因

1. 三分类覆盖了 FinanceDesk 全部数据输入场景
2. 分类边界清晰，无重叠
3. 每类数据有明确的维护责任人和更新方式
4. 便于前端展示数据来源标记，帮助用户判断数据可信度

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 需在关键模型上增加 `source_type` 字段（参考 BudgetAdjustment 的已有实现） |
| API | 新增字段需在 Schema 中暴露 |
| 页面 | 前端需展示数据来源标记 |
| ERP | 导入数据自动标记 `source=erp` |
| Dashboard | 需展示数据来源 |

### 后续不得违反事项

1. 新增任何字段时必须同时标注数据来源
2. 不得出现来源不明确的数据
3. 已有模型中未标记来源的字段在后续改造中补充
4. 导入逻辑必须自动填充来源标记，不可依赖人工选择

## BADR-006：Dashboard

### 背景

Dashboard 是 FinanceDesk 的核心价值输出，但存在边界模糊的风险。需明确 Dashboard 的只读分析定位，禁止其成为录入入口。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | Dashboard 允许部分录入 | 边界模糊，容易功能蔓延 ❌ |
| B | **Dashboard 纯只读** | 严格锁定分析层定位 ✅ |
| C | Dashboard 作为管理后台 | 承担数据管理职责 ❌ 违反单一职责 |

### 最终决策

> **Dashboard 为只读分析层。禁止数据录入、状态维护、人工修改统计指标。所有指标均来源于业务数据自动计算。**

允许读取的数据：
- 合同数据（项目、供应商合同）
- 订单数据
- 流水数据（收入、成本）
- 回款/支付数据
- ERP 同步数据
- 系统计算结果（利润、Gap、应收账龄）

### 决策原因

1. Dashboard 的设计初衷是「看」而不是「写」
2. 允许 Dashboard 录入会导致业务数据分散，维护困难
3. 数据分析结果的可信度依赖于数据源的洁净性
4. 已有独立的业务模块（项目/订单/流水）作为录入入口

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无 |
| API | 无 |
| 页面 | 无（现有 Dashboard 页面无录入功能） |
| ERP | 无 |
| Dashboard | 严格只读，维持现有 4 个端点 |

### 后续不得违反事项

1. Dashboard 页面禁止出现表单输入、数据提交按钮
2. Dashboard 禁止修改数据库中的任何记录
3. Dashboard 禁止触发任何数据录入流程
4. Dashboard 禁止维护状态（不是状态机）
5. 新增 Dashboard 端点时，必须为只读查询

- [Business_Rules.md](./Business_Rules.md) 第 5 节（Dashboard 规则）
- [03_Business_Architecture.md](./03_Business_Architecture.md) 第 4.7 节

## BADR-007：成本供应商模型

### 背景

供应商模块当前存在多个模型（Supplier, SupplierPrice, SupplierYearPrice, SupplierContract, SupplierUnitPrice），关系复杂。需要重新定义供应商的业务定位。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 传统 CRM 模式 | 管理供应商全生命周期信息 ❌ 超出需求 |
| B | **年度成本供应商合同库** | 聚焦年度框架合同 + 单价体系 ✅ |
| C | 仅保留名称和合同号 | 过于简单，无法支持年度单价管理 ❌ |

### 最终决策

> **Supplier 定义为年度成本供应商合同库，不是传统 CRM 供应商管理。**

| 属性 | 说明 |
|------|------|
| 一条记录 | 一个供应商的一个年度框架合同 |
| 合同编号 | 唯一标识 (contract_no) |
| 合同名称 | 框架合同名称 |
| 所属年度 | 年度字段 |
| 有效期 | 开始/结束日期 |
| 单价体系 | 普工/技工/高级技工/特种作业/综合单价 |

### 决策原因

1. FinanceDesk 不需要传统 CRM 的供应商全生命周期管理（资质、评级、绩效）
2. 核心需求是管理年度合同价格，用于订单成本核算
3. 一条记录 = 一个合同 ≠ 一个供应商维护多个年度，避免数据冗余
4. 现有模型（supplier + supplier_unit_price + supplier_contract）已覆盖

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无 |
| API | 无 |
| 页面 | 供应商页面维持现有设计 |
| ERP | 供应商数据可来自 ERP 同步 |
| Dashboard | 供应商相关指标基于合同库聚合 |

### 后续不得违反事项

1. 不得将供应商模块扩展为 CRM（客户管理）
2. 一条记录必须对应一个合同，不允许"一个供应商一个记录多个年度"
3. 年度变化必须创建新合同记录
4. 供应商名称唯一，同一法律主体不可重复创建

- [Business_Rules.md](./Business_Rules.md) 第 6 节（供应商规则）
- [04_Data_Model.md](./04_Data_Model.md) 第 2.2 节（供应商模块）

## BADR-008：Excel 标准

### 背景

Excel 导入导出是 FinanceDesk 的数据录入和输出主渠道。需统一模板标准，确保所有模块行为一致。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 按需开发，各模块不同 | 模板不一致，学习成本高 ❌ |
| B | **所有模块统一模板标准** | 导入/导出模板保持字段一致 ✅ |
| C | 仅导入支持，不提供导出 | 功能不完整 ❌ |

### 最终决策

> **所有业务模块必须支持统一模板的导入和导出。导入模板列头与导出列头保持一致。**

| 模块 | 导入模板 | 导出要求 |
|------|---------|---------|
| 项目 | ✅ 项目导入模板.xlsx | 全部/筛选导出 |
| 供应商 | ✅ 供应商导入模板.xlsx | 全部/筛选导出 |
| 订单 | ✅ 订单导入模板.xlsx | 全部/筛选导出 |
| 收入流水 | ✅ 收入流水导入模板.xlsx | 全部/筛选导出 |
| 成本流水 | ✅ 成本流水导入模板.xlsx | 全部/筛选导出 |
| 回款 | ✅ 回款导入模板.xlsx | 全部/筛选导出 |
| 付款 | ✅ 付款导入模板.xlsx | 全部/筛选导出 |
| 合同 | ✅ 合同导入模板.xlsx | 全部/筛选导出 |
| 单价 | ✅ 单价导入模板.xlsx | 全部/筛选导出 |

### 决策原因

1. Excel 是用户（财务人员）最熟悉的办公工具，降低上手成本
2. 统一模板标准减少导入失败率和用户困惑
3. 双向（导入+导出）形成数据闭环
4. 9 种模板已实现，标准化成本已投入

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无 |
| API | 无 |
| 页面 | 无 |
| ERP | 无 |
| Dashboard | 无 |

### 后续不得违反事项

1. 新增任何业务模块时必须同时提供导入和导出模板
2. 导入模板列头必须使用中文，与导出列头一致
3. 模板位于 `backend/templates/` 目录，集中管理
4. 不支持 Excel 导入导出的模块不得进入正式环境

## BADR-009：单次录入原则

### 背景

系统存在多个数据入口，容易导致同一业务事实被重复录入。需要建立"单次录入"规则，杜绝重复录入。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 允许重复录入，系统去重 | 浪费人力，依赖去重逻辑 ❌ |
| B | **单次录入原则** | 同一事实只能录入一次 ✅ |
| C | 自由录入，不做约束 | 数据质量无法保证 ❌ |

### 最终决策

> **同一业务事实只能录入一次。ERP 已存在的数据不得再次人工录入。FinanceDesk 应通过关联、计算、匹配完成数据整合。**

| 场景 | 录入方式 | 禁止行为 |
|------|---------|---------|
| ERP 已有项目 | 自动同步 | 禁止手工创建同名项目 |
| ERP 已有订单 | 自动同步 | 禁止手工创建同名订单 |
| 手工台账数据 | Excel 导入 | 禁止自动导入后再手工补录 |
| 系统计算结果 | 自动计算 | 禁止人工修改计算结果 |

### 决策原因

1. 重复录入是财务对账的最大风险来源
2. ERP 数据权威性高于手工录入
3. 单次录入减少人工错误
4. 通过关联和计算替代重复录入，提升系统价值

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 需通过 UNIQUE 约束 + 去重逻辑确保 |
| API | 导入接口需检查重复 |
| 页面 | 录入时提示已存在的数据 |
| ERP | 同步逻辑需正确处理 upsert |
| Dashboard | 无 |

### 后续不得违反事项

1. 导入逻辑中必须包含去重机制（ON CONFLICT DO NOTHING）
2. 前台录入必须检查数据是否已存在
3. 同一业务事实在系统中只能有一条权威记录
4. 已存在的数据如需更新，应走"更新"而非"重新创建"

### 引用文档

- [Business_Rules.md](./Business_Rules.md) 第 4.3 节（不重复记账）
- [06_ERP_Integration.md](./06_ERP_Integration.md) 第 3.1 节

## BADR-010：系统自动推导原则

### 背景

部分指标当前由人工维护或计算，存在不一致和出错风险。需明确哪些指标必须由系统自动计算，禁止人工维护。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 全部指标人工计算维护 | 效率低、易出错、对账困难 ❌ |
| B | **部分系统计算 + 部分允许人工** | 边界模糊，缺少约束 ❌ |
| C | **系统自动计算所有派生指标** | 一致性高，数据可信 ✅ |

### 最终决策

> **以下数据必须由系统自动计算，禁止人工维护计算结果：**

| 指标 | 计算方式 | 备注 |
|------|---------|------|
| Gap（差额） | `开票金额合计 - 回款金额合计` | 按订单/按项目维度 |
| 利润 | `收入合计 - 成本合计` | 按订单维度 |
| 回款率 | `已回款 ÷ 应回款` | 百分比 |
| 成本率 | `已支付 ÷ 应付` | 百分比 |
| 预算执行率 | `实际支出 ÷ 预算` | 按项目预算 |
| Dashboard 全部指标 | 自动聚合 | project-summary / profit / ar-aging |

### 决策原因

1. 人工计算一致性差，不同用户计算结果可能不同
2. 系统自动计算保证数据可信度和一致性
3. 派生指标全部基于原始数据，可追溯、可复核
4. 减少人工维护工作量

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无 |
| API | 无 |
| 页面 | 计算结果展示为只读 |
| ERP | 无 |
| Dashboard | 所有指标来源明确 |

### 后续不得违反事项

1. 派生指标的计算逻辑不得在多个地方重复实现（DRY 原则）
2. 系统计算结果不得提供手工修改入口
3. 计算逻辑变更时必须追溯所有依赖该指标的地方
4. 新增任何聚合指标时必须明确计算规则

### 引用文档

- [Business_Rules.md](./Business_Rules.md) 第 5.5 节（Gap 计算规则）
- [Business_Rules.md](./Business_Rules.md) 第 3.2 节（系统自动计算）

## BADR-011：AI 开发原则

### 背景

FinanceDesk 的开发由 AI Agent（Hermes/ChatGPT/Claude 等）主导。不同 AI 对系统理解不同，导致输出不一致。需建立统一的 AI 阅读顺序和行为规则。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 不约束 AI 阅读行为 | 不同 AI 理解不一致 ❌ |
| B | **强制阅读顺序 + BADR 约束** | 所有 AI 遵循统一的上下文语境 ✅ |
| C | 仅依赖 .hermes.md 锚点文件 | 信息量不足，易遗漏关键规则 ❌ |

### 最终决策

> **所有 AI 在开发前必须按以下顺序阅读 Knowledge 文档：**

```

Business_Rules.md         → 业务规则（本文档）

04_Data_Model.md          → 数据模型与 ER 图
（其余 Knowledge 文档按需阅读）
```

> **任何开发不得违反已确认的 BADR。**

### 决策原因

1. AI 缺少对系统历史决策的认知，BADR 提供了完整的决策背景
2. 强制阅读顺序确保 AI 先理解"为什么做"再理解"做什么"
3. 规则前置降低了 AI 产生错误输出的概率
4. BADR 不可违反的特性防止 AI 随意推翻已确认的架构决策

### 影响范围

| 维度 | 影响 |

| API | 无 |
| 页面 | 无 |
| ERP | 无 |
| Dashboard | 无 |
| **AI 开发流程** | **所有 AI 必须遵循 BADR 约束** |

### 后续不得违反事项

1. 新 AI 进入项目时必须按顺序阅读 Knowledge 文档
2. 修改代码前必须确认不违反任何 BADR
3. 新增 BADR 必须更新本阅读顺序

## BADR-012：成本供应商合同库定位

### 背景

BAF 阶段结束后，业务审计发现成本供应商模块的定位不够明确。Supplier 被模糊地理解为"供应商管理"，未区分 Master Data 与 Transaction Data 的边界。

需要明确：成本供应商合同库属于基础资料（Master Data），不是业务流程节点。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 传统 CRM Supplier | 管理供应商全生命周期 ❌ 超出经营分析范围 |
| B | **年度成本供应商合同库（Master Data）** | 年度合同 + 单价体系，仅被业务流程引用 ✅ |
| C | 合并入成本执行模块 | 基础资料与业务耦合 ❌ 违反单一职责 |

### 最终决策

> **Supplier 模块定位为：年度成本供应商合同库。属于基础资料（Master Data）。不是传统 CRM Supplier，不是业务流程节点。**

- 每一条记录代表一份年度合同，不是供应商主体
- 所有单价属于合同，不是供应商主体
- 合同库只负责维护基础资料，不录成本、不录付款、不参与经营流程
- 业务流程（成本执行、付款）仅通过 supplier_id 引用合同库

### 决策原因

1. FinanceDesk 不需要传统 CRM 的供应商全生命周期管理
2. 核心需求是管理年度合同价格，用于成本核算
3. 基础资料与业务数据分离，避免耦合
4. 一条记录 = 一份合同 ≠ 一个供应商维护多个年度
5. 单价体系随合同变更，不随供应商变更

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无（现有 supplier 等表在 BDD 阶段重构，但模型定位不变） |
| API | BDD 阶段：合同库 API 应只提供基础资料维护，不提供成本/付款功能 |
| 页面 | BDD 阶段：合同库页面只展示基础资料，不嵌入成本/付款录入 |
| ERP | 合同库不参与 ERP 同步（ERP 同步的是项目/订单/流水） |
| Dashboard | 合同库不直接贡献 Dashboard 指标（通过成本引用间接贡献） |

### 后续不得违反事项

1. 不得在成本供应商合同库中提供成本录入功能
2. 不得在成本供应商合同库中提供付款录入功能
3. 不得在成本供应商合同库中维护流程状态
4. 成本执行只能引用合同库，不能在其中维护合同信息
5. 年度变化必须创建新合同记录，不在原合同上修改年度
6. Supplier 模块在任何 UI/API 中均不得命名为"供应商管理"，应表述为"成本供应商合同库"

### 引用文档

- [Business_Master_Data.md](./Business_Master_Data.md) §2（成本供应商合同库）
- [Business_Constitution.md](./Business_Constitution.md) R008
- [Business_Rules.md](./Business_Rules.md) §6（供应商规则）

## BADR-013：Business Search 为平台统一能力

### 背景

F5（Business Search）开发完成后，确认搜索能力不应归属于单个业务模块（订单中心）。

搜索是横切关注点，所有模块都需要。

### 可选方案

| 方案 | 描述 | 评价 |
|------|------|------|
| A | 各模块自实现查询 | 重复逻辑，接口不一致 ❌ |
| B | **统一 Search API，平台级复用** | 单一入口，全部模块复用 ✅ |

### 最终决策

> **Business Search 为平台统一能力（Business Infrastructure）。所有业务模块必须复用统一 Search API。禁止各模块自行实现查询逻辑。**

### 决策原因

1. 合同中心、订单中心、收入、成本、ERP、Dashboard 都需要查询能力
2. 各自实现导致查询接口不一致，前端需对接多套 API
3. 统一 Search API 支持多维度 AND 组合，灵活性远高于各自实现
4. 可扩展性强（经营维度、ERP 维度可后续补充）

### 影响范围

| 维度 | 影响 |
|------|------|
| 数据库 | 无 |
| API | `GET /api/v1/search/business` 为统一入口 |
| 页面 | 统一搜索组件复用 |
| ERP | ERP 维度查询使用同一 API |
| Dashboard | Dashboard 查询使用同一 API |

### 后续不得违反事项

1. 模块新增时必须使用统一 Search API，不得自建查询端点
2. 新增查询维度时必须先在 Business_Search_Model.md 中定义
3. Business Search 作为平台能力，归属 Business Infrastructure
4. 前端的搜索组件也必须统一，不得各页面自建搜索栏

### 引用文档

- [Business_Search_Model.md](./Business_Search_Model.md)
- [Business_Search_Review.md](./Business_Search_Review.md)

## BADR-014：经营数据优先原则

### 背景

FinanceDesk 存在三层数据（ERP、经营、分析），职责边界不清晰。

### 可选方案

| 方案 | 描述 | 评价 |
|:----:|------|:----:|
| A | FinanceDesk 全权管理所有数据 | 与 ERP 冲突 ❌ |
| B | **三层分离：ERP 保存事实，FinanceDesk 保存经营过程，Dashboard 只负责分析** | 职责清晰 ✅ |

### 最终决策

> **ERP 保存事实。FinanceDesk 保存经营过程。Dashboard 只负责分析。三者职责永久冻结。**

| 层级 | 职责 | 维护方式 |
|:----:|------|---------|
| ERP | 保存业务事实（发票、回款、付款） | 自动同步，只读 |
| FinanceDesk 业务层 | 保存经营过程（开票阶段、预计回款、责任人） | 人工维护 |
| Dashboard | 分析展示（利润、Gap、回款率） | 系统自动计算 |

### 决策原因

1. ERP 关注财务事实，不关注经营过程（开票阶段、预计回款等）
2. FinanceDesk 的经营字段 ERP 不覆盖，必须由人工维护
3. Dashboard 的分析结果来自前两层的聚合计算
4. 三层分离避免数据混乱和重复维护

### 影响范围

| 维度 | 影响 |
|:----:|------|
| 数据库 | 经营字段独立（business_date, expected_collection_date, business_owner） |
| API | 经营字段与财务字段分离 |
| 页面 | 经营字段在收入管理页面维护 |
| ERP | ERP 不覆盖经营字段 |
| Dashboard | 引用经营字段进行分析 |

### 后续不得违反事项

1. ERP 数据在 FinanceDesk 中始终为只读
2. 经营字段（M 来源）ERP 不同步
3. Dashboard 的分析指标全部自动计算
4. 三层数据互不覆盖，互不冲突

### 引用文档

- [Revenue_Operation_Model.md](./Revenue_Operation_Model.md)
- [Revenue_Data_Source.md](./Revenue_Data_Source.md)
