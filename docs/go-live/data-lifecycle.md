# Data Lifecycle — FinanceDesk V1

> PS-002 文档，定义 FinanceDesk 的完整数据生命周期，从 ERP 导出到经营分析的端到端流程。面向财务操作人员和系统设计人员。

---

## 1. 端到端数据流

```
ERP 系统
    │
    ▼
① 导出（财务手动）
    │
    ▼
② 上传至 Sandbox（暂存，不写业务表）
    │
    ▼
③ 校验 + 匹配（自动 + 人工确认）
    │
    ▼
④ 确认导入（写入业务表）
    │
    ├──→ IncomeFlow（收入流水）
    ├──→ CostFlow（成本流水）
    ├──→ Collection（回款记录）
    ├──→ Payment（付款记录）
    │    AND
    ├──→ 更新 Order 经营状态
    ├──→ 更新 Project 经营摘要
    │
    ▼
⑤ Summary Engine 聚合
    │
    ▼
⑥ Dashboard + Workbench 展示
```

---

## 2. 角色与职责

| 角色 | 负责人 | 关键操作 |
|------|--------|----------|
| **财务操作员** | 财务部 | ERP 导出、文件上传、数据校验、人工匹配、确认导入 |
| **项目经理** | 项目一线 | 订单录入、订单信息确认、成本确认 |
| **采购专员** | 采购部 | 供应商录入、成本合同管理、报价维护 |
| **企业负责人** | 总经理/副总 | Dashboard 查看、经营分析决策 |

## 3. 月度关账流程（标准操作）

### 每月 1–5 日

```
Day 1-2: 财务导出上月 ERP 数据
         ├── 导出收支明细（income_expense）
         ├── 导出应收明细（receivable）
         └── 导出收款明细（collection）

Day 2-3: 导入 FinanceDesk
         ├── 上传 3 个 Sheet 到 Sandbox
         ├── 在 Sandbox 检查总数和金额
         └── 处理未匹配记录（人工归集）

Day 3-4: 确认导入
         ├── 确认导入 → 写入业务表
         ├── Dashboard 数据自动更新
         └── 核对 Dashboard 与 ERP 月份数字

Day 5:   关账确认
         ├── 财务确认上月数据完整
         ├── 项目经理确认订单状态正确
         └── 生成月度经营报表
```

### 月中间流程

```
日常操作:
├── 新签合同 → 录入合同信息（财务）
├── 新订单 → 录入订单信息（项目经理）
├── 新供应商 → 录入供应商信息（采购）
├── 新发票 → 随 ERP 导出导入
└── 新收款 → 随 ERP 导出导入
```

## 4. 数据状态流转

### 4.1 流水数据（核心经营数据）

```
ERP 系统
    │ 状态: 外部系统，FinanceDesk 只读不写
    ▼
ERPStagingFlow (Sandbox)
    │ 状态: pending（待匹配）/ auto_matched / manual_matched
    │ 生命周期: 导入 → 匹配 → 确认 → 写入
    ▼
IncomeFlow / CostFlow / Collection / Payment
    │ 状态: 正常（is_deleted = false）
    │ 生命周期: 写入 → 更新 → 逻辑删除（30天恢复期）
    ▼
Summary Aggregation
    │ 实时计算，无状态
    ▼
Dashboard / Workbench
    │ 只读展示
```

### 4.2 主数据（合同/订单/供应商）

```
合同 (Project)
    录入 → 进行中 → 已完成 → 已终止

订单 (Order)
    录入 → 待开票 → 正常 → 已完成

供应商 (Vendor/Supplier)
    录入 → 合作中 → 暂停合作 → 已终止
```

## 5. 数据版本管理

### 5.1 导入版本

每次 ERP 导入生成唯一的 `ImportBatch` 记录：

| 字段 | 示例 | 说明 |
|------|------|------|
| batch_no | `IMP-20260705-001` | IMP-{YYYYMMDD}-{NNN} |
| total_count | 150 | 本次导入总条数 |
| success_count | 145 | 成功导入条数 |
| failed_count | 2 | 格式错误条数 |
| duplicate_count | 3 | 重复跳过条数 |
| manual_match_count | 10 | 人工匹配条数 |

### 5.2 数据回滚

| 阶段 | 回滚能力 |
|------|----------|
| Sandbox（未确认） | ✅ 可一键清除，重新导入 |
| 已确认写入业务表 | ❌ 不支持自动回滚，需人工修正 |
| 已关账月份 | ❌ 禁止回滚，以审计跟踪方式修正 |

### 5.3 审计跟踪

系统自动记录所有关键操作的审计日志：

| 操作 | 记录内容 |
|------|----------|
| ERP 导入 | 导入人、时间、文件、条数、匹配数 |
| 人工匹配 | 操作人、时间、原始字段、目标字段 |
| 确认导入 | 导入人、时间、batch_no |
| 数据修改 | 操作人、时间、修改前、修改后 |

## 6. 数据一致性校验

### 6.1 月度对账

每月关账后必须执行以下校验：

```
① 总数对账
   导入条数 = 成功 + 失败 + 重复

② 金额对账
   ERP 导出合计 ≈ FinanceDesk 导入合计
   （允许 ±0.01 的精度误差）

③ 业务对账
   Dashboard 本月收入 ≈ ERP 本月收入
   Dashboard 本月回款 ≈ ERP 本月回款

④ 未闭合项
   待匹配记录数 = 0（关账前必须全部处理）
```

### 6.2 异常处理

| 异常 | 处理方式 |
|------|----------|
| 金额不匹配 | 检查是否有遗漏 Sheet 或重复导入 |
| 大量未匹配 | 检查关键词映射表是否需要更新 |
| 重复数据过多 | 检查导出范围是否包含了已导入的数据 |

## 7. 数据归档

### 7.1 归档策略

| 数据 | 在线保留 | 归档触发 | 归档方式 |
|------|----------|----------|----------|
| ERPStagingFlow | 90 天 | 确认导入后 +90 天 | 导出为 .json，清理 |
| 经营流水 | 永久 | — | 不归档 |
| 订单数据 | 永久 | — | 不归档 |
| 合同数据 | 永久 | — | 不归档 |
| 审计日志 | 1 年 | 超过 1 年 | 导出为 .csv |
| 导入批次 | 永久 | — | 不归档 |

### 7.2 月末备份

每月关账后自动执行：

```
① 导出全量 Project/Order/IncomeFlow/CostFlow/Collection/Payment 为 JSON
② 按年月命名: fd_backup_{YYYYMM}.json
③ 存储在服务器 /var/backups/financedesk/ 目录
④ 保留最近 12 个月的备份
```
